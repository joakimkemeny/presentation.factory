What is WebSockets?

What is WebSockets?

WebSockets is a way to solve this problem since WebSockets can be used to send commands in both directions. So the server can send information to the client even if the client haven't asked for it.

What is WebSockets?

HTML5
Firewall friendly

And WebSockets is part of all the new cool technologies that is part of HTML5. This means that it's available in most modern browsers. To only exception is the default Android browser but you can use Chrome or Firefox instead. The protocol have been stable since april so implementers are starting to catch up.

So this means no plugins, so no Flash and no Java.

It's also very firewall friendly since a socket connection is initiated over a normal http request which is upgraded to a WebSocket.

But the main reason why this is so great for running a factory is...

What is WebSockets?

Full duplex
2 bytes overhead

...that it's bidirectional and uses full duplex so we can send messages in both directions at the same time. This is of course necessary to be reliable in this kind of environment.

And the second thing is that it adds minimal overhead. Just a start and a stop byte for each message and if you compare this to the normal HTTP headers that adds around 1000 bytes it's a huge deal.

Let's look at how this can be implemented.

WebSocketServlet (Jetty)

There is no official standard for Java yet. They are working on it through JSR-356 but so far we are left in the hands of the different application server vendors. Both Glassfish and Tomcat have there own implementation but we use Jetty for our solution.

The Jetty implementation is based on the Servlet API but instead of doing a doGet we do a doWebSocketConnect when the client requests a WebSocket connection. The only thing we need to do in the servlet is to return a WebSocket to the client. We can return different WebSockets depending on the protocol but in our case we always return the same type and we pass it a reference to a WebSocketManager which is a utility class that we have created.

So what does this WebSocketClient look like.

WebSocketClient (Jetty)

First of all it implements a OnTextMessage interface. WebSockets can be both binary and text based and we use text.

The interface only have 3 methods. First onConnect where we store the actual connection to the client and then we add the client to a list of active clients. You will see why we need that later but let's just say that it's pretty useful to keep a list of all connected clients.

Next we have onClose where we just deregister the client.

The final method in the interface is onMessage and this method gets called when the client sends a message to the server over the WebSocket. In our case we delegate the parsing and handling of that message to our utility manager.

Finally we've also added a sendMessage method here which just sends a message over the connection that we stored earlier. This is the method that we use to send messages from the server to the client.

So this message, what does that look like? That's up to you but we've created a simple JSON protocol that we use and it looks like this.

Protocol

So here we have the message for our orderCreated event.

We have specified a sub protocol that we use to be able to send different kind of messages over the same WebSocket and in the command we have specified the type of the event.

Finally we have the actual data of the event and here we have the details for the created order.

So what is Backbone?

Backbone.js

Models / Collections

Backbone have three parts that work together. First we have the models and collection of models. This is where you hold the data on the client side. Orders and customers are some examples of models.

These models and collections trigger events when they are changed. So we know when there is a new order, when an order is changed or when an order is deleted.

Backbone.js

Models / Collections
Views

Next we have views. Views are responsible for showing the models and other information on the screen.

And since the models trigger events the views can listen to these events and react to them in some way.

Finally we have routers which are kind of the controllers in the MVC-model. The routers are used to map URL to a state in the application by getting the correct models and showing them in the correct views.
